Orbitir

Building a Real-Time Collaborative Kanban System: Architecture, Security, and Scalability Challenges

Java Spring Boot Angular PostgreSQL WebSocket
Leonardo Moreira Published January 2025

Abstract

Orbitir is a full-stack Kanban application designed for collaborative task management. This article explores the architectural decisions, security implementations, and optimization strategies employed in building a production-ready system that handles real-time updates, user authentication, and data persistence at scale.

1. Introduction & Motivation

The Kanban methodology, originating from Toyota's lean manufacturing system in the 1940s, has become a cornerstone of modern software development workflows. Orbitir was conceived as an exploration of how to build a collaborative Kanban system that balances simplicity with the technical rigor required for production environments.

"Orbitr - Seu espaço de produtividade: transformando ideias em realidade."

— Project Mission Statement

The project addresses three primary objectives: (1) providing an intuitive drag-and-drop interface for task management, (2) enabling real-time collaboration between team members, and (3) implementing enterprise-grade security without sacrificing user experience.

2. System Architecture

2.1 Backend Architecture

The backend follows a layered architecture pattern, implemented with Spring Boot 3.x running on Java 21. The system is organized into distinct layers:

  • Controller Layer: RESTful API endpoints with proper HTTP semantics
  • Service Layer: Business logic and transaction management
  • Repository Layer: JPA/Hibernate data access with Spring Data
  • Model Layer: Domain entities with Jakarta Persistence annotations

}

2.2 Frontend Architecture

The Angular application (v18+) leverages modern patterns including:

  • Signals: Fine-grained reactivity replacing traditional RxJS patterns for local state
  • Standalone Components: Eliminating NgModule boilerplate
  • Zoneless Change Detection: Using provideZonelessChangeDetection() for improved performance
  • Server-Side Rendering: Angular SSR with hydration support

2.3 Data Model

The domain model centers around Boards, Tasks (Tarefas), and Notes (Notas). Each task can have multiple notes with visibility controls (public/private) and read tracking for collaborative awareness.

Database Choice: PostgreSQL

PostgreSQL was selected for its robust ACID compliance, excellent JSON support for flexible metadata storage, and strong performance characteristics for concurrent read/write operations typical in collaborative applications.

3. Security Implementation

Security is implemented at multiple layers, following the defense-in-depth principle recommended by OWASP and NIST guidelines.

3.1 Authentication & Authorization

The system uses JWT (JSON Web Tokens) for stateless authentication, with tokens stored client-side and validated on each request:

Java SecurityConfig.java 
@Configuration
@EnableMethodSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .cors(cors -> {})
            .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .headers(headers -> headers
                .contentSecurityPolicy(csp -> csp.policyDirectives(
                    "default-src 'self'; " +
                    "script-src 'self' 'unsafe-inline' https://cdnjs.cloudflare.com; " +
                    "style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; " +
                    "font-src 'self' https://fonts.gstatic.com; " +
                    "img-src 'self' data:; " +
                    "object-src 'none'; " +
                    "frame-ancestors 'none';"
                ))
                .contentTypeOptions(withDefaults())
                .frameOptions(frame -> frame.deny())
            );
        return http.build();
    }
}

3.2 Security Challenges & Solutions

Challenge: XSS Vulnerability with localStorage

Storing JWT tokens in localStorage exposes them to Cross-Site Scripting (XSS) attacks. If an attacker injects malicious JavaScript, they could steal authentication tokens.

Recommended Solution: Migrate to HttpOnly cookies with short-lived access tokens (10-15 minutes) and implement refresh token rotation. The project documentation includes a detailed security roadmap addressing this concern.

Challenge: CSRF Protection with Cookies

When transitioning to cookie-based authentication, Cross-Site Request Forgery becomes a concern.

Solution: Enable Spring Security's CSRF protection with SameSite=Strict cookie attribute, combined with anti-CSRF tokens for state-changing operations.

3.3 Email Verification System

User registration requires email verification with 6-digit codes, implemented using Gmail SMTP with App Passwords for secure credential management:

  • Rate limiting on verification attempts
  • Time-limited code validity (10 minutes)
  • Brute-force protection with exponential backoff

4. Real-Time Communication

Collaborative features require instant synchronization across connected clients. The system implements WebSocket connections for real-time updates.

4.1 Implementation Pattern

The frontend uses a dedicated TaskRealtimeService that manages subscriptions and broadcasts changes:

constructor() {
    // Register realtime handlers
    this.realtimeUnsubscribes.push(

            const currentTaskId = this.task()?.id;
            if (note?.tarefa?.id === currentTaskId) {

}

5. Performance Optimization

5.1 Frontend Optimizations

  • Zoneless Angular: Removes Zone.js overhead, reducing bundle size by ~10KB and eliminating unnecessary change detection cycles.
  • Signal-based State: Fine-grained reactivity ensures only affected components re-render, compared to traditional Observable patterns.
  • Lazy Loading: Route-based code splitting for faster initial load.

5.2 Backend Optimizations

  • Connection Pooling: HikariCP for efficient database connection management.
  • JSON Serialization: @JsonIgnoreProperties to prevent circular reference issues and reduce payload size.
  • Lazy Loading: Hibernate lazy loading for associated entities with proper fetch strategies.
Java NotaTarefa.java 
@Entity
@Table(name = "notas_tarefa")
@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})
public class NotaTarefa {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @NotBlank
    @Size(max = 2000)
    private String conteudo;
    
    @ManyToOne(fetch = FetchType.LAZY)
    private Tarefa tarefa;
    
    @ManyToMany(fetch = FetchType.LAZY)
    private Set<Usuario> usuariosQueVisualizaram = new HashSet<>();
}

6. Technical Challenges Encountered

Challenge: Hibernate N+1 Query Problem

Loading tasks with their notes initially resulted in N+1 queries, severely impacting performance with large datasets.

Solution: Implemented @EntityGraph annotations and optimized JPQL queries with explicit JOIN FETCH clauses for predictable query patterns.

Challenge: State Synchronization Race Conditions

Concurrent updates from multiple users could result in stale data being displayed or lost updates.

Solution: Implemented optimistic locking with version fields and conflict resolution strategies for user-facing merge decisions.

Challenge: Cross-Platform Responsive Design

The Kanban board's drag-and-drop functionality needed to work seamlessly across desktop and mobile devices.

Solution: Utilized touch event polyfills and adaptive layouts with Tailwind CSS breakpoints, with column stacking on mobile viewports.
